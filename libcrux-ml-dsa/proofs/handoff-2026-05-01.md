## Hard rules learned this session

- **AP-7 (new): SMTPat that reveals raw `%` above the trait is
  dangerous.** Above-trait code reasons about polynomial-level
  arithmetic; raw `(v x) % q` terms in Z3's ground context cost
  non-linear arithmetic budget on every subsequent query and risk
  quantifier instantiation cascades in 32-chunk × 8-lane array
  contexts. Categorize lane_post bodies before adding SMTPat:
  - **Drop SMTPat** if body has raw `%` or `*` (e.g.,
    `reduce_lane_post`, `mont_mul_lane_post`, `shift_left_then_reduce_lane_post`).
  - **Keep SMTPat** if body exposes only Hacspec function calls
    (e.g., `decompose_lane_post`, `compute_hint_lane_post`,
    `use_hint_lane_post`, `power2round_lane_post`).
  - **Keep SMTPat** if body is plain bounds (e.g.,
    `is_binary_array_8`, `is_pos_array`).
  - When dropping an SMTPat, search for `assert (P args)` call
    sites — those expected the SMTPat to fire and need an explicit
    `lemma_*_lookup` invocation.
- **Hint regeneration cost** when invalidating a high-fanout module
  (e.g., `traits/specs.rs`): pay ~1-4 hours one-time on the heaviest
  module (Portable.Ntt or Invntt). Subsequent runs use stable hints
  and are at-or-below baseline. Don't panic when you see a 25× wall
  regression on a single module after invalidation.
