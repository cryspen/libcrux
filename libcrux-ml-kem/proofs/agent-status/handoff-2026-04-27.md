## Acceptable vs unacceptable agent outcomes (HARD RULE)

**Established this session via the B and E experiences.**

### Unacceptable

**Admit-driven scaffolding** — runs where the agent adds post-condition
citations to functions but discharges them entirely via:
- admitted bridge axioms in a helper file (agent B's `HacspecBridge.fst`
  pattern), OR
- admitted Tier-N commute lemmas in `Hacspec_ml_kem.Commute.Chunk.fst`
  + body `assume`s for invariant gaps (agent E's Phase 7a pattern), OR
- any other layered admit chain that produces zero net verification content.

**Why unacceptable**: the work appears successful (everything verifies,
posts cite the canonical hacspec functions) but no proof has actually
been performed.  All admits stack and have to be discharged by a
follow-up specialist; the agent's run delivers only structural changes
that an editor with regex could produce in 10 % of the wall time.

These patterns were tolerated this session via brief stop-conditions
("admit-with-comment if it doesn't go in 25 min") but the result is
B+B' HELD, E HELD — neither merged.  The policy was wrong.

### Acceptable

For each target an agent attempts:
1. **Land at least one real proof** before any admit-with-comment.  If
   the recipe works, demonstrate it on the simplest target (no
   cherry-picking — the *first* target attempted should aim for a
   closed proof).
2. **Skip-with-comment** is acceptable for genuinely stubborn targets
   *after* a real attempt of ~25 min (Z3 timeouts, model gaps, etc.).
   The skip's English comment must include: what was tried, the F\*
   error or the math wall, and the recommended USER follow-up.
3. **Mass-admit + post-strengthening** is acceptable ONLY when the
   admits are user-handed templates (e.g., USER-1 A8 Barrett-exact)
   that the agent is *propagating downstream*, not introducing
   themselves.

### Why the proofs SHOULD be mechanical (key principle agents missed)

The whole point of the trait-opacify design (Phases 1–5, already
landed) is that **above-trait module proofs are mechanical**.  The
trait posts in `src/vector/traits.rs::spec` already deliver per-lane /
per-branch FE-form equations behind opaque predicates.  Concretely:

- `add_post`, `sub_post`, `barrett_reduce_post`, `mont_*_post`, etc.
  give `forall_N (fun i -> opaque_lane_post v.[i] r.[i])` where the
  opaque body is a clean per-lane equation in i16-to-spec-FE form.
- An above-trait function's loop calls a trait method per chunk;
  after the loop, the compiler-supplied loop invariant + the trait
  posts at each iteration give a per-vector accumulation of opaque
  per-lane facts.
- The Tier-1 chunk-commute lemma is a `Classical.forall_intro`
  composition — the math is already in the per-lane opaque predicate
  body, so the lemma's proof is **structural, not arithmetic**.

When an agent hits the wall and reaches for body `assume`s about
"per-vector trait posts the loop produces but the bound-only invariant
doesn't carry," the agent is bypassing the trait-opacity design.  The
correct fix is to **use the trait post directly** — not to strengthen
the loop invariant manually, because the invariant the trait gives
should suffice.

If a future agent finds the trait post insufficient, that's a bug
report against `traits.rs::spec` that should escalate to the user, not
a license to introduce body `assume`s.

   data: agent A landed 4 admits in 30 min using plain `make` (50 s

   only place fstar-mcp likely helps.

   each; total < 32 GB target on this 48 GB box.  Past OOMs crashed
   the system.

   `agent/phase-N-X`.  Eager-commit logs after every meaningful step
   to `proofs/agent-status/agent-X.md` on that branch.  Defensive
   against session/machine crashes.
5. **Sleep prevention**: `caffeinate -dis` in a separate terminal
   when running long agent jobs.
6. **Bridge admits vs body proofs** (the user's #1 critique this
   session): bridge admits in `HacspecBridge.fst` are throwaway
   scaffolding because `Spec.MLKEM.*` is deleted in Phase 7l.
   Prefer body proofs when feasible.  When body proofs need

   merging admit-driven scaffolding.  E (Phase 7a) admits over

7. **Code-change policy**: Rust function bodies → don't touch
