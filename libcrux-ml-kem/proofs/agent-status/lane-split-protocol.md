# Above-trait / Below-trait lane split — ML-KEM

Modeled on the ML-DSA lane-split protocol
(`~/libcrux-ml-dsa-proofs/libcrux-ml-dsa/proofs/agent-status/lane-split-protocol.md`),
adapted for ML-KEM's heavier per-module proofs. The structure is the same;
the workload distribution differs (ML-KEM has more Z3-walled work and a
bigger above-trait surface, esp. INTT/Polynomial/Matrix).

Verification work is split across two parallel branches that meet at
the `Libcrux_ml_kem.Vector.Traits.{fst,fsti}` + `…Traits.Spec.fst` contract.

| Lane | Worktree | Branch | Verifies |
|---|---|---|---|
| **Above-trait** | `~/libcrux-ml-kem-above-trait/` (TBD by other agent) | TBD | `Polynomial`, `Ntt`, `Invert_ntt`, `Matrix`, `Serialize`, `Sampling`, `Compress`, `Ind_cpa{,.Unpacked}`, `Ind_cca{,.Unpacked,.Multiplexing,.Instantiations.*}`, `Mlkem*`, plus `Hacspec_ml_kem.Commute.{Chunk,Bridges}` and `Hacspec_ml_kem.ModQ` (the spec/bridge layer above traits) |
| **Below-trait** | `~/libcrux-trait-opacify/` (this) | `trait-opacify` | `Vector.Portable.*`, `Vector.Avx2.*`, plus the trait `.fsti` / `.fst` / `Vector.Traits.Spec.fst` themselves |

Both lanes share the same trait pre/post. The Makefile in each lane
keeps the *other* lane's modules in `ADMIT_MODULES`.

## Synchronisation protocol

1. **Trait pre/post changes are owned by the above-trait lane.** This
   branch never edits `src/vector/traits.rs`'s pre/post conjuncts, the
   `Vector::*` method signatures, or the `spec` submodule's pre/post
   helpers (`add_post`, `barrett_reduce_post`, `*_branch_post`, etc.).
   Below-trait owns:
   - Impl bodies: `src/vector/{portable,avx2}.rs` and the `*/`-subdir
     primitives (`portable/arithmetic.rs`, `portable/compress.rs`,
     `portable/ntt.rs`, `portable/serialize.rs`, `portable/sampling.rs`
     and AVX2 mirrors).
   - Below-trait commute lemmas in
     `specs/ml-kem/proofs/fstar/commute/Hacspec_ml_kem.Commute.{Chunk,Bridges}.fst`
     **only when** they're producing impl-side bridge facts.
2. **Cherry-pick the trait commit** (taking only `traits.rs` and
   `Vector.Traits.{fst,fsti}` + `Traits.Spec.fst`, never the
   above-trait Makefile changes) when the above-trait lane updates the
   contract. Track the last cherry-pick SHA below.
3. **Mismatches must be flagged early** in this file under "Open
   findings" — not silently fixed by tightening the impl post. The
   above-trait lane will rebase / amend.

## Edit-check loop (per the 2026-04-29 empirical study)

**Canonical inner loop**: plain `make check/<Mod>.fst` from
`proofs/fstar/extraction/`. ~9 s for admitted modules; full SMT for
non-admitted. Surgically `rm` only the target module's `.checked` when
iterating.

"admit everything else" Makefile override (cache-cascade traps). See
`proofs/agent-status/edit-check-loop-comparison.md` for the full report.

## Style discipline — what posts may and may not contain

Both lanes follow these rules. Above-trait lane should be especially
strict because their posts propagate into many call sites.

### Rule SD1 — Mod-q opacity

Any post (trait or above-trait helper) that expresses a mod-q residue
equation MUST use `Hacspec_ml_kem.ModQ.mod_q_eq`, not raw `x % 3329 == y % 3329`.

- ✅ Allowed: `Hacspec_ml_kem.ModQ.mod_q_eq (v r) (v vec * c * 169)`
- ❌ Forbidden in posts: `(v r % 3329) == (v vec * c * 169) % 3329`

Unfold via `lemma_mod_q_eq_unfold` should appear ONLY inside Commute
bridge lemmas (where the inner machinery still uses raw `%`); never in
above-trait posts or bodies.

Rationale: opacity prevents Z3 from doing irrelevant integer-arithmetic
quantifier instantiation in above-trait Z3 contexts. Established
2026-04-29 with `Hacspec_ml_kem.ModQ`; verified on Vector.{Portable,Avx2}.fst,
Polynomial.fst, Invert_ntt.fst.

### Rule SD2 — `forall (i: nat). i < N ==> p i` vs `Spec.Utils.forall<N>`

`Spec.Utils.forall16`, `forall8`, etc. unfold to a fixed N-conjunction
(`p 0 /\ p 1 /\ ... /\ p (N-1)`). Z3 ground-instantiates these without
firing the quantifier engine. Generic `forall (i: nat). i < N ==> p i`
triggers SMT quantifier instantiation — much heavier.

**Below-trait** (16-lane SIMD): always prefer `Spec.Utils.forall16`.
Empirically much faster (per session data: rlimit 80 with `forall16`
vs rlimit 400+ for the generic forall in identical proofs). Standardize
on this shape for all 16-lane posts.

**Above-trait** (256-lane polynomial / K-lane vector): `forall256`
and `forallK` don't exist (would be too large to expand); generic
`forall (i: nat). i < 256 ==> ...` is necessary. **Mitigate** by:
- Wrapping inner per-lane facts in opaque per-chunk predicates
  (e.g., `subtract_reduce_finalize_chunk`) so the post-loop subtype
  check sees ~16 opaque atoms instead of 256 unfolded forall facts.
- Using `Classical.forall_intro` over the loop range explicitly in
  body proofs, rather than letting Z3 instantiate the quantifier.

Rule of thumb: if the index range is ≤32, use `Spec.Utils.forall<N>`.
If larger, use generic `forall` but gate inner facts behind an opaque
wrapper.

### Rule SD3 — Trait-post opaque wrappers

Branch posts for nested-if-ladder methods (NTT layer 2 forward/inverse)
SHOULD be wrapped in `[@@ "opaque_to_smt"]` per-branch predicates
(e.g., `inv_ntt_layer_2_step_branch_post`). Established Phase 2 pattern;
proved decisive on layer 2 (which had a 2.7-min Z3 timeout pre-opacity).

### Rule SD4 — `reveal_opaque` MUST be targeted, never global, inside loop bodies

**Established 2026-04-29 (USER-13 closure, commit `ee69454dd`).** This
is the most important style discipline rule for impl-side bodies that
manipulate opaque-bounded vectors.

#### The two reveal forms

| Form | Shape | Effect |
|---|---|---|
| **TARGETED** ✅ | `reveal_opaque (\`%X.foo) (X.foo arg1 arg2 ...)` | Unfolds `foo` only at THAT specific instance; one fact added to context. |
| **GLOBAL** ❌ | `reveal_opaque (\`%X.foo) (X.foo)` | Unfolds the predicate UNIVERSALLY; every appearance of `foo` in scope becomes its underlying definition. Inside a loop body this fires *every iteration* over *every prior chunk's* opaque atom. |

#### Why the GLOBAL form is a Z3 saturation trap

The just-closed USER-13 wall (`Libcrux_ml_kem.Invert_ntt.invert_ntt_at_layer_2_`)
had

```rust
hax_lib::fstar!(
    r#"reveal_opaque (`%Libcrux_ml_kem.Vector.Traits.Spec.is_i16b_array_opaque)
                (Libcrux_ml_kem.Vector.Traits.Spec.is_i16b_array_opaque)"#
);
```

(GLOBAL form) inside a `for round in 0..16` loop. `is_i16b_array_opaque
n a` is defined as `forall i. i < Seq.length a ==> is_i16b n a.[i]`.
The global reveal substitutes that definition into EVERY appearance of
`is_i16b_array_opaque` in scope — i.e., 16 chunks worth of bound facts
turned into 16 universal forall facts, *each iteration*. Q108 (the
iter-end loop-invariant subtyping) saturated rlimit 200 in 42 s, 400
in 75-110 s, 800 hung past 12 min. The fix was to **delete the global
reveal entirely** and replace it with a single targeted `assert (...)`
of the per-iteration bound, which the trait post already carries
opaquely.

#### When you actually need a reveal

You need a reveal ONLY when the goal mentions the opaque predicate AND
Z3 needs the underlying definition to discharge that specific
instance. In practice for `is_i16b_array_opaque` and similar bound
predicates, this is almost never inside a loop body — the loop
invariant carries the bound atoms opaquely and the trait posts
produce them opaquely.

#### Operational checklist when writing impl-side bodies

1. Default to NO `reveal_opaque` calls in loop bodies. Add one only
   when verification fails.
2. If verification fails inside a loop, FIRST try a targeted
   `assert (P specific-args)` to inject the needed atom. Only if that
   doesn't suffice, add a **targeted** `reveal_opaque (\`%P) (P
   specific-args)`.
3. Reach for the GLOBAL form **only at top-level** in lemma proofs
   that intend to operate on the predicate's definition (e.g.,
   `is_bounded_vector_higher` in `Polynomial.Spec.fst`). Even then,
   keep the lemma small.
4. When you see an existing global reveal inside a loop body — even
   if the function currently passes — flag it as a candidate for
   removal. The current pass may rely on a brute-force
   `--z3rlimit 800 --split_queries always`. Removing the reveal
   often lets you drop to `--z3rlimit 100`.

#### Audit-time signature to look for

Any line that matches the regex `reveal_opaque\s*\([`%][^)]+\)\s*\(\s*[A-Z][\w.]+\s*\)`
inside a `hax_lib::fstar!` block in `*.rs` or inside a `let` body
in an extracted `.fst` is the GLOBAL form and is suspect. The
specific case of `(X.foo)` (no arguments) at the second position is
the smoking gun.

## Coordination conventions

- **Single-direction trait edits**: above-trait lane edits, below-trait
  cherry-picks. Never merge both directions.
- **Hot files** (touched by both lanes for different reasons):
  - `Hacspec_ml_kem.Commute.Chunk.fst` — both lanes add lemmas here.
    Convention: append to clearly-marked sections (e.g.,
    `(* Phase 7c additions *)`); never edit existing lemma bodies.
    Below-trait additions are bridges from impl posts to spec; above-trait
    additions are spec-level commute lemmas.
  - `Hacspec_ml_kem.Commute.Bridges.fst` — same convention.
  - `Hacspec_ml_kem.ModQ.fst` (NEW, 2026-04-29) — opaque mod-q
    abstraction. Below-trait owns the definitions + intro/unfold lemmas;
    above-trait consumes via the public interface.
- **Cherry-pick frequency**: above-trait cherry-picks below-trait's
  Commute.Chunk additions weekly (or whenever a finding is resolved).
  Below-trait cherry-picks above-trait's traits.rs immediately on each
  contract change.

## Why both protocols exist

ML-DSA's lane-split protocol is the original (mature; battle-tested
with findings F-1 through F-7). ML-KEM mirrors the structure but the
workload differs:

- ML-DSA: smaller per-module proofs; mostly mechanical commute lemmas;
  primary risk is post-shape mismatches (signed vs unsigned,
  centered vs non-negative).
- ML-KEM: larger Z3 contexts (NTT layer 1 4-zeta wall, USER-7 record
  bridge, sample_matrix_A K-quantification); primary risk is Z3
  saturation and SMT-context pollution.

The same `traits.rs ↔ commute lemmas ↔ admit-the-other-lane` pattern
applies. The findings will look different — likely more "Z3 wall" type
findings on ML-KEM than the algebraic mismatches that dominate ML-DSA.
